# PRD: ИИ-Агент «Виртуальный менеджер по продажам» — EdPalm

**Версия:** 1.0
**Дата:** 2026-03-02
**Статус:** Draft → Review

---

## 1. Контекст и цель

EdPalm — онлайн-школа семейного образования (1–11 классы). Отдел продаж вручную ведёт пролонгации, квалификацию лидов и дожим сделок. Это дорого, медленно и непредсказуемо.

**Цель V1:** Запустить ИИ-агента, который самостоятельно закрывает типовые сделки и пролонгации без участия менеджера — 24/7, в едином интерфейсе, с интеграцией в CRM и платёжную систему.

**Метрики успеха V1:**
| Метрика | Цель |
|---|---|
| Конверсия в оплату (пролонгация) | ≥ 40% от диалогов |
| Конверсия в оплату (новый лид) | ≥ 15% от диалогов |
| Доля диалогов без эскалации на менеджера | ≥ 70% |
| Время ответа агента | ≤ 5 сек |
| Uptime | 99.5% |

## 4. Аутентификация и идентификация пользователя

Агент работает в трёх контекстах входа:

**4.1. Портал (авторизованный пользователь)**
- Портал (PHP) генерирует короткоживущий JWT (TTL 15 мин) с `user_id`, `name`, `phone`
- JWT передаётся в React-приложение как `?token=...` при открытии здания-магазина
- Python-бэкенд верифицирует JWT → получает профиль пользователя из DMS по `user_id`

**4.2. Telegram Mini App**
- Telegram передаёт подписанный `initData`
- Бэкенд верифицирует подпись через HMAC-SHA256 с `BOT_TOKEN`
- Ищет пользователя в Supabase по `telegram_id` → подтягивает профиль из DMS

**4.3. Внешняя ссылка (WhatsApp / лендинг)**
- Генерируется одноразовый подписанный токен в URL (`?t=...`)
- TTL токена: 48 часов
- Неавторизованный режим: агент работает как с новым лидом, квалифицирует с нуля

## 5. Функциональные требования (V1)

### 5.1. Сценарий: Новый лид

**Триггер:** пользователь открыл агента впервые (неизвестный контакт)

**Флоу:**
1. Приветствие, представление агента
2. Квалификация: имя → кол-во детей → класс(ы) → цель (учёба, ЕГЭ, доп.курсы)
3. Подбор релевантного продукта из базы знаний (RAG)
4. Презентация оффера с ценой
5. Обработка возражений
6. Отправка ссылки на оплату (DMS API)
7. При оплате: вебхук → агент отправляет подтверждение + создаёт сделку в amoCRM
8. При неоплате: follow-up через 24ч, 48ч, 7 дней

**Действия агента:**
- `search_knowledge_base(query)` — поиск в RAG
- `generate_payment_link(product_id, amount, client_data)` — DMS API
- `create_lead_amocrm(contact, deal_data)` — amoCRM API
- `escalate_to_manager(reason, chat_history)` — передача диалога

---

### 5.2. Сценарий: Пролонгация

**Триггер:** CRM-событие (за 30 дней до окончания периода) → агент инициирует диалог

**Флоу:**
1. Персонализированное приветствие (знает имя, класс, тариф)
2. Подведение итогов года / периода (опционально — данные из LMS)
3. NPS-опрос (1 вопрос: «Как оцените обучение от 1 до 10?»)
4. Предложение следующего периода с акцентом на выгоды
5. Оффер со скидкой раннего бронирования
6. Обработка возражений
7. Ссылка на оплату
8. Follow-up цепочка при неоплате

**Дополнительные действия агента:**
- `get_student_progress(student_id)` — через DMS API (Moodle обёрнут внутри DMS)
- `get_deal_data(contact_id)` — amoCRM API
- `update_deal_stage(deal_id, stage)` — amoCRM API

---

### 5.3. Сценарий: Кросс-сейл / Апсейл

**Триггер:** в ходе диалога или отдельная рассылка

**Доп. продукты:** Ключи Познания, Логика, Профориентация, Разговорные клубы

**Флоу:**
1. Агент определяет релевантный доп. продукт по профилю клиента
2. Органично вплетает предложение в диалог
3. Презентация выгод
4. Оффер + ссылка на оплату

---

### 5.4. Guardrails (ограничения агента)

| Ситуация | Действие агента |
|---|---|
| Запрос скидки которой нет в базе | «Уточню у менеджера» → эскалация |
| Вопрос о конкретном преподавателе | Отвечает только по данным из базы знаний |
| Клиент явно негативит | Немедленная эскалация с историей |
| Клиент просит человека | Немедленная эскалация |
| Вопрос вне компетенции | «Давайте я соединю вас с менеджером» |
| Попытка prompt injection | Игнорирование, ответ в рамках роли |

---

### 5.5. Эскалация на менеджера

При эскалации:
1. Агент завершает сообщение: «Соединяю вас с менеджером, он ответит в ближайшее время»
2. В amoCRM: создаётся/обновляется сделка, прикрепляется полная история диалога
3. Менеджер получает уведомление в Telegram с контекстом
4. Диалог «замораживается» — агент не отвечает, ждёт закрытия тикета

## 6. База знаний (RAG)

**Источники (загружаются при старте и обновляются вручную):**
- Описания продуктов по классам (1–11): форматы, цены, расписание
- FAQ — частые вопросы клиентов
- Скрипты лучших менеджеров (возражения и ответы)
- Регламенты и правила школы

**Технология:**
- Документы → чанкинг → OpenAI `text-embedding-3-small` → pgvector в Supabase
- При запросе: similarity search top-5 → вставка в system prompt

## 7. Интеграции

### 7.1. amoCRM (azaprimemat)
- OAuth 2.0 авторизация (токены хранятся в Supabase, auto-refresh)
- **Читаем:** контакт по телефону/telegram_id, активная сделка, тариф, сумма
- **Пишем:** создание контакта, создание/обновление сделки, смена этапа воронки, примечание с историей диалога
- Воронки: Sales (`10490514`), Service (`10490518`)

### 7.2. DMS API
Go-бэкенд, REST на порту 8080. Исходники изучены (`/dms-main`).
- Auth: `POST /v1/api/auth` → JWT (access 10 мин + refresh). Агент использует сервисную учётку EdPalm.
- Поиск клиента: `POST /v1/api/contacts/search` — по телефону или email
- Создание заказа: `POST /v1/api/orders`
- Платёжная ссылка: `POST /v1/api/payment/link` — Tochka Bank (уже интегрирован в DMS)
- Подтверждение оплаты: `POST /v1/api/payment/confirm` — вебхук от банка
- Данные студента / успеваемость: через DMS (Moodle обёрнут внутри)

### 7.3. LMS (Moodle)
Прямая интеграция с Moodle **не нужна** — DMS уже содержит синхронизацию с Moodle и предоставляет данные об успеваемости через свой API. Отдельный клиент для Moodle в агенте не разрабатывается.

## 8. Каналы и UI

### 8.1. React Chat UI (ядро)

Standalone SPA, встраивается в любую оболочку.

**Компоненты:**
- `ChatWindow` — лента сообщений со стримингом (SSE)
- `MessageInput` — поле ввода + кнопка отправки
- `QuickReplies` — быстрые кнопки-ответы (опционально)
- `TypingIndicator` — индикатор «агент печатает»
- `PaymentCard` — карточка с кнопкой «Оплатить» (встроена в сообщение)
- `EscalationBanner` — баннер при передаче на менеджера

**UX-требования:**
- Стриминг ответов (как ChatGPT) через SSE
- Адаптив: mobile-first
- Поддержка светлой и тёмной темы (для Telegram Mini App берётся из `colorScheme`)
- История диалога сохраняется между сессиями

### 8.2. Telegram Mini App
- Тот же React SPA
- Авторизация через `initData`
- Кнопка «Оплатить» → открывает платёжную ссылку в браузере
- Поддержка Back Button Telegram

### 8.3. Портал (здание-магазин)
- Тот же React SPA встраивается через `<iframe>` или рендерится напрямую
- Портал передаёт JWT при открытии
- Агент знает кто вошёл, персонализирует с первого сообщения

## 10. Нефункциональные требования

| Требование | Значение |
|---|---|
| Время ответа | ≤ 5 сек до первого токена (стриминг) |
| Доступность | 24/7, uptime ≥ 99.5% |
| Безопасность | JWT verification, HMAC для Telegram, подписанные URL-токены, HTTPS only |
| Хранение данных | Диалоги в Supabase, PII в соответствии с 152-ФЗ |
| Язык | Русский (основной). Архитектура готова к мультиязычности (EN/AR для ОАЭ) |
| Tone of Voice | С родителями — уважительно и экспертно. Со старшеклассниками — дружелюбно |
