## 1. Workspace Switcher

### Slack

**Паттерн:** Вертикальный sidebar слева с иконками воркспейсов (аватарки).

**Как работает:**
- Каждый воркспейс -- отдельная иконка в левом вертикальном баре
- Клик по иконке мгновенно переключает весь контекст (каналы, DM, уведомления)
- Бейджи с числом непрочитанных сообщений на иконках воркспейсов
- Нет анимации перехода -- переключение мгновенное (instant swap)
- На мобильном -- свайп вправо открывает список воркспейсов

**Горячие клавиши (Mac):**
- `Cmd+Shift+S` -- развернуть/свернуть switcher
- `Ctrl+Shift+Tab` / `Ctrl+Tab` -- предыдущий/следующий воркспейс
- `Cmd+[число]` -- конкретный воркспейс
- `Cmd+K` -- быстрый переход к каналу/беседе (Quick Switcher)

**Техническая архитектура:** Slack перестроил десктоп-клиент с отдельного Electron-процесса на каждый воркспейс к единому процессу. Каждый воркспейс получил свой Redux store (`данные, статус подключения, WebSocket для realtime`), а context management реализован через redux-thunk -- Redux сам управляет контекстом воркспейса, без глобальных переменных и без явной передачи workspace ID в каждую функцию.

**Дизайн-принцип:** Контекстная изоляция. Переключение воркспейса полностью меняет контекст -- каналы, участники, настройки. Нет смешения данных между воркспейсами.

### Discord

**Паттерн:** Вертикальная полоса серверов слева (Server Switcher).

**Как работает:**
- Иконки серверов -- круглые аватарки в вертикальном списке
- Активный сервер выделяется pill-индикатором слева (белая полоска)
- Hover -- иконка становится квадратной с скруглением, tooltip с названием
- Бейджи непрочитанных -- белая точка (есть непрочитанные) или число (упоминания)
- Разделитель между DM-зоной (вверху) и серверами
- Drag & drop для переупорядочивания серверов
- Folder-группировка серверов (вложенные папки)
- Внизу -- кнопка "+" для добавления/создания сервера

**Анимация:** Morph-эффект при hover (круг -> скругленный квадрат, ~200ms ease). Pill-индикатор анимированно появляется/уезжает.

**Мобильная версия:** Свайп вправо открывает список серверов. Тот же визуальный язык, но адаптирован под touch-зоны.

**Дизайн-принцип:** Пространственная память. Пользователь запоминает позицию сервера в списке. Drag & drop и папки -- инструменты для организации.

---

### Notion

**Паттерн:** Dropdown-switcher в верхней части sidebar.

**Как работает:**
- Имя текущего воркспейса + иконка (эмодзи/аватар) -- в шапке sidebar
- Клик открывает dropdown со списком всех воркспейсов аккаунта
- Каждый воркспейс -- имя + эмодзи + количество участников
- Переключение мгновенное (полная смена дерева страниц в sidebar)
- "Create or join a workspace" внизу dropdown
- Каждый воркспейс имеет собственные настройки (имя, иконка, домен)

**Мобильная версия:** Тот же паттерн -- tap на имя воркспейса в шапке sidebar открывает bottom sheet со списком.

**Дизайн-принцип:** Минимальный overhead. Switcher занимает минимум места, переключение -- в 2 клика. Фокус на контенте, а не навигации.

---

### Microsoft Teams

**Паттерн:** Multi-account switcher через аватар пользователя.

**Как работает:**
- Аватар пользователя в верхнем правом углу
- Клик открывает dropdown с аккаунтами/тенантами
- Переключение между organization tenants (не workspace в терминах Slack)
- Левый sidebar: Activity, Chat, Teams, Calendar -- всегда видны
- Teams-каналы раскрываются внутри sidebar
- Notification badges для каждого аккаунта

**Дизайн-принцип:** Enterprise-ориентированный. Акцент на security boundaries между организациями.

### Обобщение паттерна Workspace Switcher

| Продукт | Расположение | Визуал | Переключение | Контекст |
|---|---|---|---|---|
| Slack | Вертикальный бар слева | Иконки-аватарки | Instant swap | Полная изоляция |
| Discord | Вертикальный бар слева | Круглые иконки + pill | Instant + morph | Полная изоляция |

| Teams | Аватар справа вверху | Dropdown аккаунтов | Reload-like | Организационная изоляция |

## 2. ChatGPT: переключение между GPTs

### Текущий паттерн (2025-2026)

**Sidebar:** Левый sidebar с историей чатов. В шапке -- model picker (dropdown: GPT-4o, GPT-4o mini, o1, o3, и т.д.).

**GPT Store и Custom GPTs:**
- Доступ к GPT Store через sidebar или "Explore GPTs"
- Каждый кастомный GPT -- отдельная "личность" с аватаркой, именем, описанием
- Выбор GPT начинает НОВЫЙ чат (не переключение внутри существующего)
- При выборе GPT -- аватарка GPT отображается вместо дефолтной иконки ChatGPT
- Системный промпт GPT скрыт от пользователя

**@-mentions (mid-2024+):**
- Внутри чата можно тегнуть `@GPT-name` для вызова кастомного GPT
- Это запускает ответ от конкретного GPT внутри текущего чата
- Визуальный индикатор: аватарка GPT рядом с ответом
- Контекст предыдущего чата доступен, но GPT отвечает по своим инструкциям

**Model Picker:**
- Dropdown в шапке чата
- Смена модели НЕ начинает новый чат -- можно переключить mid-conversation
- Визуальный индикатор текущей модели всегда виден

**Дизайн-принцип:** Разделение "личности" (GPTs) и "мощности" (модели). Личность -- новый чат. Мощность -- inline switch.

**Релевантность для Эврики:**
- Роли (Продавец/Поддержка/Учитель) ближе к GPTs -- это смена "личности"
- Но у нас роли привязаны к зданиям на портале, а не к выбору пользователя
- @-mention паттерн может быть полезен если нужно вызвать "экспертизу" другой роли

## 3. Telegram: меню и режимы ботов

### Паттерн Menu Button

**Стандартное поведение:**
- У каждого бота в чате есть Menu Button (слева от поля ввода)
- По умолчанию показывает список команд бота с описаниями
- Можно кастомизировать -- показывать разные команды разным пользователям
- Альтернативно -- запуск Web App вместо меню команд

**Дизайн-принцип:** Один бот -- одна точка входа. Режимы внутри бота, а не несколько ботов.

### Reply Keyboards

**Как работает:**
- Бот может заменить стандартную клавиатуру на кастомные кнопки
- Нажатие кнопки = мгновенная отправка текста в чат
- `one_time_keyboard` -- клавиатура скрывается после нажатия
- Подходит для выбора режима: "Продавец", "Поддержка", "Учитель"

**Паттерн:**
```
[Продавец]  [Поддержка]  [Учитель]
```
Пользователь нажимает -- бот переключает режим.

### Inline Keyboards

**Как работает:**
- Кнопки прикреплены к конкретному сообщению (под сообщением)
- Нажатие НЕ отправляет сообщение в чат -- вызывает callback
- Можно обновлять кнопки и текст сообщения без отправки нового
- Идеально для: настроек, пагинации, toggles

**Best Practice от Telegram:**
> "Consider editing your keyboard when the user toggles a setting button or navigates to a new page rather than sending new messages -- faster and smoother UX."

**Дизайн-принцип:** Inline edit вместо новых сообщений. Обновление UI на месте создает ощущение приложения, а не чата.

### Mini Apps (Web Apps)

**Как работает:**
- Полноценное веб-приложение внутри Telegram
- Запускается из Menu Button или inline-кнопки
- Доступ к данным пользователя (имя, ID, фото)
- Может быть full-screen или half-screen (MainButton внизу)
- Оптимизация графики на основе устройства пользователя

**Релевантность для Эврики:** Наш Mini App -- это как раз Web App в Telegram. Переключение ролей можно реализовать внутри Mini App (React UI), а не через Telegram-команды.

## 4. WhatsApp Business: бот-агент хэндофф

### Паттерн Bot-to-Human Handoff

**Общепринятый паттерн в индустрии (WhatsApp, Intercom, Zendesk, Kommunicate):**

#### Триггеры переключения

1. **Сложность запроса** -- бот понимает, что не может ответить
2. **Запрос пользователя** -- кнопка "Поговорить с человеком"
3. **Анализ настроения** -- бот определяет фрустрацию пользователя
4. **Критичность проблемы** -- разделение информационных запросов и инцидентов
5. **Супервизия агента** -- человек подтверждает ответ перед отправкой

#### Три фазы хэндоффа

**Pre-handoff:**
- Actionable кнопка "Перевести на менеджера"
- Уведомление пользователю: "Переключаю вас на специалиста..."
- Сбор контекста для передачи агенту

**Wait:**
- Позиция в очереди + ETA
- Опция оставить email-тикет если ожидание слишком долгое
- Маршрутизация: Round Robin (FIFO) или контекстная (по языку, теме, географии)

**Post-handoff:**
- Передача ПОЛНОЙ истории чата агенту
- Пользователь НЕ повторяет информацию
- Агент видит все, что бот уже узнал

**Дизайн-принцип:** Бесшовность и сохранение контекста. Пользователь не должен повторяться.

### WhatsApp Business Session Rules

- 24-часовое окно для ответов на сообщение пользователя
- Template messages для outbound (нужно pre-approve)
- Interactive messages: кнопки, списки, product cards
- Нет визуального разделения "бот ответил" vs "человек ответил" в стандартном WhatsApp

**Важное ограничение:** В стандартном WhatsApp нет UI-индикатора, отличающего бота от человека. Это решается на уровне формулировок ("Привет! Я виртуальный помощник..." vs "Здравствуйте, меня зовут Мария...").

## 5. Tab/Pill навигация в чатах

### Apple Messages: iMessage vs SMS

**Паттерн:** Цветовое кодирование вместо табов.
- Синие пузыри = iMessage (Apple-to-Apple)
- Зеленые пузыри = SMS/MMS/RCS
- Нет табов для фильтрации -- все сообщения в одном потоке
- Визуальная дифференциация достаточна для различения типов

### Типы табов (по NN/g)

**In-page Tabs:**
- Организуют контент внутри одной страницы
- Переключение мгновенное (нет загрузки)
- Один таб выбран по умолчанию
- Подходит для: фильтрация чатов, переключение режимов

**Navigation Tabs:**
- Навигация между разными страницами/экранами
- Широкий scope -- несвязанный контент
- Может быть задержка загрузки
- Подходит для: вкладки "Чаты"/"Звонки"/"Контакты"

**КРИТИЧЕСКОЕ правило:** Нельзя смешивать in-page и navigation tabs в одном контроле. Это дезориентирует пользователя.

### Segmented Controls (Apple HIG / Material Design)

**Что это:** Горизонтальная группа кнопок равной ширины для переключения между связанными видами/режимами.

**Когда использовать:**
- 2-5 взаимоисключающих вариантов
- Переключение между видами одного контента
- Мгновенный результат (без Submit)

**Визуал:** Capsule-shaped фон, активный сегмент выделен заливкой. Анимация slide при переключении (~200ms).

**Пример для Эврики:**
```
[Продавец]  [Поддержка]  [Учитель]
```
Pill/segmented control в шапке чата. Активная роль -- заполненный сегмент, остальные -- outline.

### Chips (Material Design)

**Filter Chips:** Для включения/выключения фильтров. Множественный выбор.
**Choice Chips:** Для выбора одного из вариантов. Единственный выбор.

**Для Эврики:** Choice Chips подходят для переключения ролей -- один активный чип, остальные неактивны.

### Best Practices для табов (NN/g)

1. **Минимум 2 индикатора выбранного таба**: подчеркивание + жирность, или заливка + цвет
2. **Невыбранные табы должны быть видны** -- плохой контраст = "невидимые" опции
3. **Только один ряд табов** -- стекание создает хаос
4. **Табы сверху контента** -- внизу или сбоку пользователи пропускают
5. **1-2 слова на таб** -- для сканируемости
6. **Высокоприоритетный контент -- первый таб по умолчанию**
7. **Mixed-case текст** -- не ALL CAPS (снижает читаемость)

## 6. Образование и геймификация

### Duolingo Max (AI Features)

**Персонажи-тьюторы:**
- Lily, Junior, Eddy, Zari -- каждый со своей личностью
- Lily -- саркастичный подросток, Eddy -- восторженный друг, Zari -- стильная путешественница
- Выбор персонажа для Role Play и Video Call

**Режимы AI:**
- **Explain My Answer** -- AI объясняет, почему ответ правильный/неправильный (запускается кнопкой после упражнения)
- **Role Play** -- диалоговая симуляция с AI-персонажем (отдельный экран, полноэкранный чат)
- **Video Call** -- AI-видеозвонок с персонажем (полноэкранный, камера персонажа + текст)
- **Adventures** -- нарративные квесты с выбором

**Переключение режимов:**
- Из основного потока уроков -- кнопка на карточке (не tab, а action card)
- Каждый режим -- отдельный полноэкранный экран
- Назад -- через навигационную стрелку или завершение сессии
- Нет табов между режимами -- каждый режим погружающий (immersive)

**Дизайн-принцип:** Immersive mode. Каждый режим -- отдельный мир. Нет параллельного доступа к другим режимам. Уменьшает cognitive load.

### Khan Academy (Khanmigo)

**Роли AI:**
- Tutor -- помогает решать задачи, не давая прямой ответ
- Debate Partner -- оппонент в дискуссии
- Writing Coach -- помогает с эссе и текстами
- Teacher Assistant -- помогает учителю создавать материалы

**Переключение ролей:**
- Роль определяется контекстом входа (из какого упражнения/урока пользователь пришел)
- Нет явного UI для переключения ролей -- бот автоматически адаптируется
- Activity-based routing: "Help me write an essay" = Writing Coach mode

**Дизайн-принцип:** Контекстный автоматизм. Роль определяется действием, а не явным выбором пользователя.

## 7. Figma, Linear, Notion: анимации

### Figma

**File/Project Switching:**
- File browser -- grid/list с превью файлов
- Double-click открывает файл в новой вкладке браузера или в десктоп-приложении
- Вкладки файлов -- как в браузере (horizontal tabs)
- Quick Actions (`Cmd+/`) -- поиск по всем командам

**Workspace Switching:**
- Dropdown в верхнем левом углу с именем workspace
- Список teams/projects внутри sidebar
- Transition: мгновенный swap контента

**Анимация:** Figma использует `cubic-bezier(0.8, 0, 0.2, 1)` с длительностью ~400ms для основных переходов. Минималистичные fade/slide при навигации.

### Linear

**Workspace/Team Switching:**
- Sidebar слева с командами и проектами
- Keyboard-first navigation (`Cmd+K` для любого действия)
- Мгновенные переходы между views (Issues, Cycles, Projects)
- Breadcrumb-навигация сверху

**Анимация:** Быстрые fade transitions (~150-200ms). Фокус на скорости, не на эффектах. "Craft in the details, speed in the experience."

**Дизайн-принцип:** Keyboard-first + speed. Каждое действие доступно через клавиатуру. Переходы быстрые.

### Notion

**Page/Workspace Navigation:**
- Sidebar с деревом страниц (collapsible)
- Breadcrumb сверху при навигации вглубь
- Cmd+K -- Quick Find (поиск по всему workspace)
- Переключение workspace -- dropdown в шапке sidebar

**Анимация:** Мягкие fade (~200ms). Content morphing при переключении страниц. Sidebar smooth collapse/expand.

## 8. Принципы дизайна

### UX-законы применительно к переключению ролей

**Закон Хика:** Время принятия решения растет с количеством вариантов. Три роли -- оптимально (не перегружает).

**Закон Миллера:** Человек удерживает 7+-2 элементов в рабочей памяти. При переключении ролей контекст предыдущей роли быстро теряется -- нужно сохранять и восстанавливать состояние.

**Закон Якоба:** Пользователи ожидают, что ваш продукт работает как другие знакомые продукты. Если пользователь знаком со Slack/Discord -- ожидает workspace-подобный switcher.

### Принципы анимации (NN/g)

**Четыре функции анимации:**

1. **Обратная связь на микро-действия** -- подтверждение, что система приняла действие
2. **Переходы между состояниями** -- анимация помогает понять смену режима (morph иконки, slide контента)
3. **Навигационный контекст** -- zoom/slide показывают, где пользователь в иерархии
4. **Сигнификаторы взаимодействия** -- направление анимации подсказывает доступные жесты

**Критическое правило:** Анимация должна быть ненавязчивой, короткой и тонкой. Избыток анимации создает когнитивную перегрузку.

### Визуальная индикация режима (NN/g)

**Паттерн из статьи о модах:**
- Сильные визуальные сигналы: разный фон, акцент на активном индикаторе, изменение курсора
- Избыточность (redundancy) -- используйте несколько индикаторов одновременно (цвет + подчеркивание + иконка)
- Четко названные режимы с текстовыми лейблами
- Tooltips, объясняющие эффект режима
- Если ошибка режима опасна -- избегайте режимов вообще

**Пример для Эврики:** При переключении на "Поддержку" -- меняется цвет header, иконка, приветственное сообщение. Тройная индикация: цвет + иконка + текст.

### Микро-взаимодействия

**Определение:** Пара "триггер-обратная связь". Триггер -- действие пользователя или изменение состояния системы. Обратная связь -- узконаправленное визуальное изменение.

**Для переключения ролей:**
- Триггер: клик по табу/pill роли
- Обратная связь: pill анимированно смещается, цвет header плавно меняется, аватар роли появляется с fade-in, приветственное сообщение роли fade-in

## 9. Доступность (Accessibility)

### ARIA-паттерн для табов (W3C WAI)

**Обязательные роли:**
- `role="tablist"` -- контейнер для табов
- `role="tab"` -- каждый таб
- `role="tabpanel"` -- контент каждого таба

**Обязательные атрибуты:**
- `aria-controls` -- на табе, ссылка на связанный tabpanel
- `aria-selected="true/false"` -- активный/неактивный таб
- `aria-labelledby` -- на tabpanel, ссылка на связанный таб
- `aria-orientation="horizontal/vertical"` -- ориентация tablist

**Клавиатурная навигация:**
- `Tab` -- перемещает фокус в tablist (на активный таб), затем выходит
- Стрелки влево/вправо -- переключение между табами (с циклическим обходом)
- `Space`/`Enter` -- активация таба
- `Home`/`End` -- первый/последний таб
- `Delete` -- удаление таба (если поддерживается)

**Два режима активации:**
- **Автоматическая:** Таб активируется при получении фокуса (рекомендуется если контент грузится быстро)
- **Ручная:** Пользователь явно нажимает Space/Enter (если контент тяжелый)

**Рекомендация W3C:** Автоматическая активация предпочтительна -- ручная `"significantly hampers users' ability to navigate efficiently across the tab list"`.

### Для Эврики

```html
<div role="tablist" aria-label="Выбор роли агента">
  <button role="tab" aria-selected="true" aria-controls="panel-seller" id="tab-seller">
    Продавец
  </button>
  <button role="tab" aria-selected="false" aria-controls="panel-support" id="tab-support">
    Поддержка
  </button>
  <button role="tab" aria-selected="false" aria-controls="panel-teacher" id="tab-teacher">
    Учитель
  </button>
</div>
```

## 10. Mobile vs Desktop

### Desktop

| Паттерн | Пример | Преимущества |
|---|---|---|
| Вертикальный sidebar | Slack, Discord | Всегда видны все варианты, бейджи |
| Horizontal tabs/pills | ChatGPT model picker | Компактно, привычно |
| Dropdown | Notion workspace | Минимальный footprint |
| Cmd+K Quick Switcher | Linear, Figma, Notion | Keyboard-first, быстро |

### Mobile

| Паттерн | Пример | Преимущества |
|---|---|---|
| Bottom tab bar | iOS/Android стандарт | Thumb-friendly, всегда видна |
| Swipe между экранами | Telegram, Slack | Жесто-ориентировано |
| Bottom sheet (dropdown) | Notion mobile | Не занимает экран |
| Hamburger menu | Мало кто использует | Прячет навигацию (плохо) |

### Ключевые отличия

**Touch-зоны:** Минимум 44pt (iOS) / 48dp (Android) для tap targets.

**Приоритет контента:** На мобильном экранная недвижимость критична -- `"Prioritize content over chrome"` (NN/g).

**Hamburger -- плохо:** Hamburger menu скрывает навигацию. `"Out of sight is out of mind"` (NN/g). На мобильном лучше bottom tab bar.

**Для Эврики (3 роли):** 3 элемента -- идеально для segmented control / pill navigation. Помещается и на mobile, и на desktop. На mobile -- в header чата. На desktop -- в header или sidebar.

## 11. Рекомендации для Эврики

### Контекст задачи

У Эврики три роли: Продавец, Поддержка, Учитель. Переключение сейчас привязано к зданиям на портале или параметру `agent_role`. Нужен UI-паттерн для:
1. Портала (здание -> чат)
2. Telegram Mini App
3. Внешней ссылки

### Рекомендованный паттерн: Segmented Control + Contextual Auto-Switch

**Основной компонент:** Pill/Segmented Control в шапке чата.

```
┌─────────────────────────────────────────┐
│  [Продавец]  [Поддержка]  [Учитель]     │
│─────────────────────────────────────────│
│                                         │
│  Привет! Я Эврика, ваш консультант.    │
│  Чем могу помочь?                       │
│                                         │
│  ...                                    │
│                                         │
│  [Введите сообщение...]        [Send]  │
└─────────────────────────────────────────┘
```

### Поведение переключения

**Автоматическое определение роли:**
- Вход из здания "Магазин" на портале -> автовыбор "Продавец"
- Вход из здания "Секретарь" -> автовыбор "Поддержка"
- Вход из учебного здания -> автовыбор "Учитель"
- Внешняя ссылка с параметром `?role=seller` -> автовыбор

**Ручное переключение:**
- Segmented control в шапке позволяет переключиться
- При переключении: fade transition (~200ms), смена цвета header, приветственное сообщение роли
- Контекст чата: начинается новый чат ИЛИ загружается последний чат с этой ролью

### Визуальная дифференциация ролей

| Роль | Цвет | Иконка | Приветствие |
|---|---|---|---|
| Продавец | Зеленый (#10B981) | Магазин/Корзина | "Привет! Помогу подобрать программу обучения" |
| Поддержка | Синий (#3B82F6) | Поддержка/Headset | "Здравствуйте! Помогу с организационными вопросами" |
| Учитель | Фиолетовый (#8B5CF6) | Книга/Graduation | "Привет! Давай разберем учебный материал" |

Тройная индикация: цвет header + иконка роли + текст приветствия.

### Анимация переключения

1. **Pill slide:** Активный pill анимированно перемещается к выбранной роли (~200ms ease-out)
2. **Color morph:** Цвет header плавно переходит к цвету новой роли (~300ms)
3. **Content fade:** Старый чат fade-out, новый чат fade-in (~200ms crossfade)
4. **Avatar morph:** Аватар/иконка роли -- scale-in с небольшим bounce (~250ms)

**CSS пример:**
```css
.role-pill {
  transition: transform 200ms ease-out, background-color 200ms ease-out;
}
.chat-header {
  transition: background-color 300ms ease-in-out;
}
.chat-content {
  transition: opacity 200ms ease-in-out;
}
```

### Сценарии по каналам

**Портал (здание на карте):**
- Клик по зданию -> открытие чата с автоматически выбранной ролью
- Segmented control виден, но роль предвыбрана
- Пользователь может переключиться если нужно

**Telegram Mini App:**
- Segmented control в шапке Mini App
- Или: Reply Keyboard при первом сообщении:
  ```
  Выберите, с кем хотите поговорить:
  [Консультант по обучению]  [Поддержка]  [Учитель]
  ```
- После выбора -- inline keyboard для повторного переключения

**Внешняя ссылка:**
- Роль определяется параметром `?role=seller`
- Segmented control скрыт или показывает только текущую роль
- Для новых клиентов: только продавец (упрощение выбора)

### Чего НЕ делать

1. **Не переключать роль автоматически по содержимому сообщения** -- это создаст mode errors и дезориентацию
2. **Не скрывать индикатор текущей роли** -- пользователь всегда должен знать, с кем разговаривает
3. **Не требовать подтверждения переключения** (dialog) -- это замедляет. Переключение должно быть instant
4. **Не сбрасывать контекст при переключении** -- сохранять историю чата для каждой роли
5. **Не использовать hamburger menu для ролей** -- три элемента прекрасно помещаются в pills
